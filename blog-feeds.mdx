## Overview

The personal website ships three machine-readable XML feeds alongside the human-facing blog:

- An **RSS 2.0 feed** at `/rss.xml` that lets readers subscribe to the blog in any feed reader.
- An **Atom feed** at `/atom.xml` for feed readers and tools that prefer Atom over RSS.
- A **sitemap** at `/sitemap.xml` that lists every static page, blog post, and internal project link for search engines to crawl.

All three feeds are generated from the same source of truth as the rest of the site, so new blog articles and projects flow into them automatically — there is nothing to publish by hand.

## RSS feed

### What it serves

The RSS feed is a standard RSS 2.0 channel titled **Patrick's Blog** and served at `https://paaatrrrick.com/rss.xml`. Each item corresponds to one blog article and includes:

- `title` — the article title
- `link` and `guid` — the canonical article URL at `/blog/{id}`
- `description` — the article's `summary` field, XML-escaped

Articles are emitted in the order they appear in the blog's `articles` list, so the newest post sits at the top of the feed as soon as it lands in the source.

### When to use it

Point this URL at any RSS reader (Feedly, NetNewsWire, Reeder, Inoreader, etc.) to subscribe:

```text
https://paaatrrrick.com/rss.xml
```

You can also link to it from the blog page with a standard discovery tag so browsers and readers can auto-detect it:

```html
<link rel="alternate" type="application/rss+xml" title="Patrick's Blog" href="/rss.xml" />
```

### Adding a new article

The feed is built by calling `generateRssFeed(posts)` with the same `articles` array that drives the blog UI. To add a post to the feed, add it to that array — the next build will include it automatically. Make sure each article defines at least:

- `id` — used to build the `/blog/{id}` URL
- `title` — used as the item title
- `summary` — used as the item description

If `summary` is missing the item will still render, but the description will be empty in feed readers.

## Atom feed

### What it serves

The Atom feed is a standard Atom 1.0 document titled **Patrick's Blog** and served at `https://paaatrrrick.com/atom.xml`. It mirrors the RSS feed's contents — one `<entry>` per blog article — with these fields:

- `title` — the article title
- `link` — the canonical article URL at `/blog/{id}`
- `id` — the same canonical URL, used as the entry's stable identifier
- `summary` — the article's `summary` field, XML-escaped

Entries are emitted in the same order as the `articles` list, so the newest post appears first.

### When to use it

Some readers and integrations (Mastodon's RSS-to-fediverse bridges, certain static-site aggregators, older versions of NetNewsWire) prefer or require Atom. Point them at:

```text
https://paaatrrrick.com/atom.xml
```

You can also advertise it for auto-discovery next to the RSS link:

```html
<link rel="alternate" type="application/rss+xml" title="Patrick's Blog" href="/rss.xml" />
<link rel="alternate" type="application/atom+xml" title="Patrick's Blog" href="/atom.xml" />
```

### Adding a new article

The Atom feed is built by calling `generateAtomFeed(posts)` with the same `articles` array that drives the RSS feed and blog UI, so adding an article to that list publishes it to all three feeds at once. The same minimum fields apply: `id`, `title`, and `summary`.

## Sitemap

### What it serves

The sitemap is a standard `urlset` XML document served at `https://paaatrrrick.com/sitemap.xml`. It is produced by `generateSitemap()` and contains three groups of URLs:

1. **Static pages** — the top-level routes of the site: `/`, `/blog`, `/timeline`, `/uses`, `/now`, `/reading-list`, and `/talks`.
2. **Blog posts** — one `/blog/{id}` entry per article in the blog's `articles` list.
3. **Internal project links** — every project link whose `href` starts with `/`, pulled from the `myProjects` array documented in [Projects section: structure and updates](/projects).

External links (anything starting with `http`) are skipped so the sitemap only advertises pages that live on the personal website itself.

### When to use it

Submit the sitemap URL to search engines (for example, in Google Search Console or Bing Webmaster Tools) so new posts and projects are discovered quickly.

### Referencing the sitemap from robots.txt

Crawlers also look for a `Sitemap:` directive in `robots.txt`. Add one line at the bottom of `public/robots.txt` so any crawler that fetches `robots.txt` can find the sitemap without being told where it lives:

```text public/robots.txt
User-agent: *
Disallow:

Sitemap: https://paaatrrrick.com/sitemap.xml
```

### Adding new entries

The sitemap rebuilds from live data on every generation, so you don't edit it directly:

- **New blog posts** appear automatically once they are added to the `articles` array (the same change that surfaces them in the RSS feed).
- **New project links** appear automatically when a project in `myProjects` includes a link whose `href` starts with `/`. Use a root-relative path for any case study or deep-dive that lives on the site itself.
- **New static routes** (for example, a new `/changelog` page) need to be added to the sitemap's static path list so crawlers can find them.
