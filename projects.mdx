## Overview

The projects showcase page lives at `/projects` and renders a responsive grid of side projects. Each card shows the project's name (linked to its repo), an Active or Archived status badge, a short tagline, a longer description, and a list of tech tags. Visitors can narrow the list with a free-text search box and a row of tech filter buttons.

The page is data-driven: every card is rendered from an entry in the `projects` array exported by `src/constants/projects.js`. To add, remove, or update a project, edit that array — no component changes required.

## When to use this page

- **Adding a new side project** you want to highlight to visitors.
- **Marking a project as archived** when you stop maintaining it but want to leave it discoverable.
- **Updating tech tags** so a project shows up under the right filter button.

If you only want to tweak how cards look, edit `src/components/Projects.js` and `src/components/projects.css` instead.

## Entry schema

Each entry in the `projects` array supports the following fields. All fields are required.

| Field         | Type       | Description                                                                                       |
| ------------- | ---------- | ------------------------------------------------------------------------------------------------- |
| `name`        | string     | Project name, rendered as a linked card heading.                                                  |
| `tagline`     | string     | One-line summary shown directly under the name.                                                   |
| `description` | string     | A sentence or two of detail. Searchable.                                                          |
| `status`      | string     | Either `'Active'` or `'Archived'`. Drives the badge color.                                        |
| `tech`        | string[]   | Tech tags shown on the card and aggregated into the filter bar.                                   |
| `link`        | string     | URL the card heading links to. Opens in a new tab.                                                |

### Status badges

The `status` field renders a colored pill next to the project name:

- `'Active'` — projects you're still shipping or maintaining.
- `'Archived'` — projects that are read-only or no longer maintained.

The badge class is derived from `status.toLowerCase()`, so the value must match one of those two strings exactly for the styling to apply.

### Tech tags and the filter bar

Every unique string in any project's `tech` array becomes a button in the filter bar, sorted alphabetically. An `All` button (the default) shows every project. Selecting a tech filter hides any card whose `tech` array does not include that tag.

Keep tag spelling consistent across entries — `Node` and `Node.js` produce two separate filter buttons.

## Adding a new project

Open `src/constants/projects.js` and add an object to the `projects` array. New entries appear in the order they are declared.

```js src/constants/projects.js
const projects = [
    {
        name: 'Inkwell',
        tagline: 'A markdown-first note-taking app',
        description: 'A local-first notes app with bidirectional links, a command palette, and end-to-end encrypted sync.',
        status: 'Active',
        tech: ['React', 'Rust', 'SQLite'],
        link: 'https://github.com/example/inkwell',
    },
    // ...more projects
];

export default projects;
```

<Tip>
When you archive a project, change `status` from `'Active'` to `'Archived'` rather than deleting the entry — the card stays discoverable through search and tech filters.
</Tip>

## How search and filters combine

The page has two independent controls that intersect:

1. **Search box** — matches the trimmed, lower-cased query against each project's `name`, `tagline`, and `description`. A project is kept if any of the three contains the query as a substring.
2. **Tech filter** — keeps only projects whose `tech` array includes the active tag. `All` disables this filter.

A project must pass **both** filters to be shown. If the combined result is empty, the grid renders an empty state that reads "No projects match your search."

### Writing search-friendly copy

Because search only looks at `name`, `tagline`, and `description`, make sure those fields contain the words a visitor is likely to type. For example, if a project is your "habit tracker," include the word `habit` in the tagline or description even if it's obvious from the name.

## Writing good entries

- **`tagline`** — one short phrase. Lead with the category ("Self-hosted bookmark manager") rather than marketing language.
- **`description`** — one or two sentences. Mention the core mechanic and any notable capabilities (an API, importers, encryption).
- **`tech`** — list the technologies a reader would care about, not every dependency. Three to five tags per project is a good target.
- **`link`** — point at the canonical home for the project. A GitHub repo is fine; a hosted site is better if one exists.
